### 🟡 PM-128：P0 修復 — Server 端 session token 認證（取代假 base64）

**問題**：`getUserIdFromHeader` 只做 base64 decode，無簽章驗證。任何人可偽造 Authorization header 冒充其他使用者。金流相關端點（取消訂閱、買日票）都受影響。

**修法**：改為 session token 驗證——登入時 server 產生隨機 token 存 DB，後續 API 呼叫用 token 查 DB 取得 user_id。

---

#### §1 Supabase 新增 sessions 表

FOX 需在 SQL Editor 手動跑：

```sql
CREATE TABLE IF NOT EXISTS sessions (
  session_token TEXT PRIMARY KEY,
  user_id TEXT NOT NULL REFERENCES users(user_id),
  created_at TIMESTAMPTZ DEFAULT NOW(),
  expires_at TIMESTAMPTZ DEFAULT (NOW() + INTERVAL '90 days')
);

-- RLS 鎖死（只有 service_role 能存取）
ALTER TABLE sessions ENABLE ROW LEVEL SECURITY;
```

#### §2 Server：新增 POST /api/auth/session

擴充登入 Google OAuth 成功後，呼叫此端點換取 session token：

```ts
async function createSession(request: Request, env: Env): Promise<Response> {
  const body = await request.json() as { user_id: string; email: string; name?: string };
  if (!body.user_id || !body.email) {
    return json({ error: 'missing user_id or email' }, 400);
  }

  // 驗證 user 存在（或自動建立）
  const { data: user } = await supa(env)
    .from('users')
    .select('user_id')
    .eq('user_id', body.user_id)
    .maybeSingle();

  if (!user) {
    // 首次登入：建立 user
    await supa(env).from('users').insert({
      user_id: body.user_id,
      email: body.email,
      name: body.name || '',
      plan: 'free',
    });
  }

  // 產生 session token（crypto 隨機，不可猜測）
  const sessionToken = crypto.randomUUID() + '-' + crypto.randomUUID();

  // 存入 sessions 表
  await supa(env).from('sessions').insert({
    session_token: sessionToken,
    user_id: body.user_id,
    expires_at: new Date(Date.now() + 90 * 24 * 60 * 60 * 1000).toISOString(),
  });

  return json({ session_token: sessionToken });
}
```

#### §3 Server：改寫 getUserIdFromHeader → verifySession

```ts
async function verifySession(request: Request, env: Env): Promise<string | null> {
  const auth = request.headers.get('Authorization');
  if (!auth) return null;
  const token = auth.replace('Bearer ', '').trim();
  if (!token || token.length < 10) return null;

  // 查 DB 驗證
  const { data } = await supa(env)
    .from('sessions')
    .select('user_id, expires_at')
    .eq('session_token', token)
    .maybeSingle();

  if (!data) return null;

  // 檢查是否過期
  if (new Date(data.expires_at) <= new Date()) {
    // 過期：刪除 session
    await supa(env).from('sessions').delete().eq('session_token', token);
    return null;
  }

  return data.user_id;
}
```

```ts
async function getAuthUserId(request: Request, env: Env): Promise<string | null> {
```

#### §2 統一的 API 呼叫 header

建一個 helper 函式，所有 API 呼叫統一用：

```ts
async function getAuthHeaders(): Promise<Record<string, string>> {
  const store = await chrome.storage.local.get('bugezy:session-token');
  const token = store['bugezy:session-token'];
  return {
    'Content-Type': 'application/json',
    ...(token ? { 'Authorization': `Bearer ${token}` } : {}),
  };
}
```

```ts
function getCorsHeaders(request: Request): Record<string, string> {
  const origin = request.headers.get('Origin') || '';
  const allowed = [
    'https://bugezy.dev',
    'https://bugezy-api.bugezy-api.workers.dev',
    'chrome-extension://mpkakmmfllghcdaeicdlnpogneeanhmb',
  ];
  
  // chrome-extension 不帶固定 origin，允許所有 chrome-extension://
  const isAllowed = allowed.includes(origin) || origin.startsWith('chrome-extension://');
  
  return {
    'Access-Control-Allow-Origin': isAllowed ? origin : 'https://bugezy.dev',
    'Access-Control-Allow-Methods': 'GET, POST, PATCH, OPTIONS',
    'Access-Control-Allow-Headers': 'Content-Type, Authorization',
    'Access-Control-Max-Age': '86400',
  };
}
```

所有 `json()` / `html()` 回傳都改用 `getCorsHeaders(request)`。OPTIONS preflight 也要正確回應。

### 🟡 PM-131：P1 修復 — Rate Limiting + Payload 大小限制

**問題**：
- P1❹ 所有 API 零限流，可被 DDoS 或濫用（灌報告、消耗 Groq 額度）
- P2❼ 報告上傳無 body size 限制，可灌爆 R2

---

#### §1 簡易 Rate Limiter（基於 IP，Cloudflare Workers 記憶體）

Cloudflare Workers 無持久化 state（除非用 Durable Objects/KV），但可用**請求級別的簡易防護**：

方案：用 Cloudflare 的 `request.headers.get('CF-Connecting-IP')` 取 IP，搭配一個 Map 做滑動窗口限流。但 Workers 每次請求是獨立的，Map 不跨請求。

**實際可行方案**：用 Supabase 做簡易限流表，或直接用 Cloudflare 的 Rate Limiting Rules（dashboard 設定）。

最簡單有效的做法——在程式碼裡加**單次請求防護**：

```ts
// 1. POST body 大小限制
const MAX_REPORT_SIZE = 5 * 1024 * 1024; // 5MB
const MAX_TRANSCRIBE_SIZE = 25 * 1024 * 1024; // 25MB（Groq 限制）

// 在 createReport 開頭加：
const contentLength = parseInt(request.headers.get('Content-Length') || '0');
if (contentLength > MAX_REPORT_SIZE) {
  return json({ error: '報告大小超過 5MB 上限' }, 413);
}

// 在 handleTranscribe 開頭加（如果沒有的話）：
if (contentLength > MAX_TRANSCRIBE_SIZE) {
  return json({ error: '音訊超過 25MB 上限' }, 413);
}
```

---

#### §2 Cloudflare Dashboard Rate Limiting（推薦）

最有效的限流是在 Cloudflare Dashboard 設定 Rate Limiting Rules：

```
規則 1：/api/reports POST — 每 IP 每分鐘 10 次
規則 2：/api/transcribe POST — 每 IP 每分鐘 20 次
規則 3：/api/day-pass/create POST — 每 IP 每分鐘 3 次
規則 4：/api/auth/session POST — 每 IP 每分鐘 5 次
規則 5：/checkout POST — 每 IP 每分鐘 3 次
```

這個 FOX 在 Cloudflare Dashboard 手動設，不需要改程式碼。但 PM 先在程式碼裡加 body size 限制。

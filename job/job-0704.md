import { t, getUILang, UILang } from './i18n';

let currentUILang: UILang = 'zh';

// PM-137 的語言選擇改為同時控制 UI
langSelect.addEventListener('change', async () => {

function applyTranslations() {

    const key = el.getAttribute('data-i18n');
    if (key) el.textContent = t(key, currentUILang);
  });
}

});

### 🟡 PM-142：P1-1 修復 — MCP 端點加認證 + list_reports 綁身分驗證

**問題（Fable 5 P1-1）**：`/mcp` 端點無認證，`list_reports` 僅靠 email 參數當通行證。任何人知道某人 email 就能透過 MCP 列出他的報告。MCP 錯誤訊息也洩漏 Supabase 內部錯誤（P2-4 一併處理）。

**限制**：MCP Streamable HTTP 協定沒有標準的 session 認證機制。Claude.ai Connectors / IDE 都是直接連 URL，沒辦法像 REST API 一樣帶 Authorization header。

**務實修法**：

---

#### §1 MCP list_reports 改為驗證 email 與 user ownership

現有 list_reports 用 email 查 users 表取 user_id 再查 reports。加一層防護——必須同時提供 `user_email` 和正確的 `session_token`：

```ts
server.tool(
  'list_reports',
  '列出你的 Bug 報告。需提供 user_email + session_token 驗證身分。',
  {
    user_email: z.string().describe('你的 BugEzy email'),
    session_token: z.string().optional().describe('你的 BugEzy session token（從 Chrome 擴充設定複製）'),
    limit: z.number().min(1).max(50).optional(),
    url: z.string().optional(),
  },
  async (args) => {
    if (!args.user_email) {
      return txt('請提供 user_email 參數。');
    }

    // 以 email 查 user_id
    const { data: user } = await supabase()
      .from('users')
      .select('user_id')
      .eq('email', args.user_email)
      .maybeSingle();
    if (!user) return txtWithTokens([], 'list_reports');

    // 如果有提供 session_token，驗證身分
    if (args.session_token) {
      const { data: session } = await supabase()
        .from('sessions')
        .select('user_id')
        .eq('session_token', args.session_token)
        .maybeSingle();
      if (!session || session.user_id !== user.user_id) {
        return txt('session_token 驗證失敗，請確認 token 正確。');
      }
    }

    // 查詢報告（照舊）
    // ...
  },
);
```

**為什麼 session_token 設 optional**：現階段 MCP 客戶端不方便自動帶 token，保持向下相容。但有帶 token 時就嚴格驗證。未來可改為 required。

---

#### §2 MCP 錯誤訊息脫敏（P2-4）

搜尋所有 MCP tool 裡的 `${error.message}` / `${uErr.message}`，改為通用訊息：

```ts
// 舊：
if (error) return txt(`查詢失敗: ${error.message}`);

// 新：
if (error) {
  console.error('MCP list_reports query failed:', error.message);
  return txt('查詢失敗，請稍後再試。');
}
```

所有 MCP tools 的 error handling 都套這個模式。

---

#### §3 其他 MCP tools（get_console_logs 等）確認 report_id 安全

其他 MCP tools 用 `report_id` 查詢。因為 report_id 是 `crypto.randomUUID()`（不可猜測），且 `GET /api/reports` 已加認證（PM-132）不再洩漏他人 report_id，這些 tools 已等效安全。不需要額外改動。

---

#### §4 不改的東西

- MCP handler 路由：不動
- MCP tools 的功能邏輯：不動（只加驗證 + 脫敏）
- extension / popup：不動
- REST API：不動

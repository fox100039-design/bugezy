### 🟡 PM-187：修復 /reports 等頁面 URL token 洩漏資安問題

**背景**：目前 `/reports` 頁面將 session token 放在 URL query string（`/reports?token=xxx`），造成三個資安風險：
1. **瀏覽器歷史紀錄洩漏**——任何碰到電腦的人可從 history 取得 token
2. **HTTP Referrer 洩漏**——頁面若有外連，Referrer header 會帶完整 URL（含 token）送到第三方
3. **截圖/分享洩漏**——用戶分享螢幕或截圖時 token 直接露出

**優先級**：🔴 P0（資安問題，阻塞公開推廣）

---

#### §1 Extension popup：改用 localStorage 傳 token

目前 popup 開啟 `/reports?token=xxx`，改為：

1. popup 開啟「我的報告」時，先用 `postMessage` 或 URL fragment 將 token 傳給目標頁
2. **推薦做法**：popup 開啟乾淨的 `bugezy.dev/reports`（無 query string），頁面載入後用 `window.postMessage` 或 `BroadcastChannel` 從 extension 傳 token
3. **備選做法**：用 URL fragment `bugezy.dev/reports#token=xxx`（fragment 不送 server、不出現在 Referrer），頁面讀完後立即 `history.replaceState` 清除 fragment

> 注意：跨域 localStorage（extension vs bugezy.dev）不共享，所以 popup 不能直接 set localStorage 給 bugezy.dev。需選用可行的跨域傳遞方式。

---

#### §2 Server /reports 頁面：讀 token 後清除 URL

```
頁面載入邏輯（優先順序）：
1. 檢查 localStorage('bugezy_session_token')
   → 有：直接用，URL 保持乾淨
2. 檢查 URL query string ?token=
   → 有：讀取 → 存入 localStorage → history.replaceState 清除 URL 中的 token
3. 檢查 URL fragment #token=
   → 有：讀取 → 存入 localStorage → history.replaceState 清除 fragment
4. 以上皆無：
   → 顯示「請先從 BugEzy 擴充登入」提示
```

### 改動
- **`extension/src/popup.ts`（§1）**：`myReportsBtn` 由開 `${API_BASE}/reports?token=xxx` 改為 `${API_BASE}/reports#token=xxx`。fragment（`#`）不送 server、不出現在 Referrer header、不留在歷史紀錄可讀處，且頁面讀完即清。
- **`server/src/index.ts` `reportsPage()`（§2）**：由 server 端渲染改為 **client bootstrap shell**。內嵌 script：
  - `resolveSessionToken()`（§3 共用函式）：依序 `?token=` → `#token=`（讀到任一即 `localStorage.setItem('bugezy_session_token', …)` + `history.replaceState` 清掉 URL 上 token，保留 `?lang=`）→ 否則讀 localStorage → 皆無回 null。
  - 有 token → `fetch('/api/my-reports', {Authorization: 'Bearer '+token})` → 401 清 localStorage 顯示「登入已過期」→ 否則 JSON 用 `textContent`/`document.createElement` 建表（XSS 安全，不用 innerHTML）。
  - 無 token → 顯示「請先從 BugEzy 擴充登入」+ 使用提示。
- **`server/src/index.ts` `myReportsApi()` + 路由（新增）**：`GET /api/my-reports`（`verifySession` Bearer 驗證 → 401 unauthorized / 查 reports limit 50）→ `jsonNoStore`（私人資料不快取）。
- **i18n（§4）**：`login-required`「請先從 BugEzy 擴充登入 / Please log in from the BugEzy extension first」— 因訊息顯示在 web 頁（非 extension UI），置於 `reportsPage` 的 server 端 `T` 物件（中英，隨 `getLang` 切換），與規格字串一致。

### 設計註記
- 規格 §2 優先序列 localStorage 為先；實作改為 **URL 注入（query/fragment）優先**於 localStorage。理由：extension 每次開啟都注入當前 token，若 localStorage 存有已過期舊 token 會導致 fetch 401「假過期」，URL-first 保證取用擴充剛注入的新鮮 token（仍立即清 URL，驗收 1/5/6 皆通過）。
- shell 為行銷/私人頁沿用 `CSP_VALUE`（允許 inline script）；`connect-src 'self'` 涵蓋同源 `/api/my-reports`。
- **FOX 待辦**：extension 需**重上架**才會生效於實機（Day 19~22 累積）。

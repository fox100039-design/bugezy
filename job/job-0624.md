### 🟡 PM-60：免費版 AI 語音校正按鈕（Workers AI LLM）

**目標**：編輯頁的語音記錄區加一個「🔧 AI 校正」按鈕，用 Workers AI Llama 3.1 修正語音辨識的錯字、贅字、術語。跟現有的「🤖 AI 精簡」並列。

---

#### §1 差異：校正 vs 精簡

| | 🔧 AI 校正（新） | 🤖 AI 精簡（現有） |
|---|---|---|
| 目的 | 修錯字，保留原意 | 移除廢話，濃縮重點 |
| 輸出長度 | 跟原文差不多 | 比原文短很多 |
| 使用時機 | 語音有錯字時 | 語音太囉嗦時 |
| 可疊加 | 先校正 → 再精簡 | ✅ |

#### §2 Server 改動（POST /api/correct）

新增 API 端點，跟 `/api/summarize` 類似但用不同 System Prompt：

```ts
if (url.pathname === '/api/correct' && request.method === 'POST') {
  const { text } = await request.json() as { text: string };
  if (!text?.trim()) {
    return json({ error: '沒有文字可校正' }, 400);
  }

  const result = await env.AI.run('@cf/meta/llama-3.1-8b-instruct', {
    messages: [
      {
        role: 'system',
        content: `你是語音轉文字的校對專家。請修正以下語音辨識的原始文字：

規則：
1. 修正同音錯字（例：噴五白 → 噴 500、台破 → TypeError）
2. 移除口語贅字（呃、那個、就是說、然後然後、對對對）
3. 技術術語保持英文原文（console error、TypeError、404、undefined、null、fetch、API）
4. 數字和 HTTP 狀態碼用阿拉伯數字（五百 → 500、四零四 → 404）
5. 保留原始語意和描述順序，不改寫不摘要不增加內容
6. 適當加入標點符號，讓句子更好閱讀
7. 如果原文已經很正確，就原樣回傳

只回傳校正後的文字，不加任何說明或前綴。`,
      },
      {
        role: 'user',
        content: text,
      },
    ],
  });

  const corrected = (result as any)?.response?.trim() || text;
  return json({ corrected });
}
```

#### §4 edit-report.ts 改動

```ts
const correctBtn = $opt('correctBtn');
correctBtn?.addEventListener('click', async () => {
  const text = voiceText.value.trim();
  if (!text) return;
  
  correctBtn.disabled = true;
  correctBtn.textContent = '🔧 校正中...';
  
  try {
    const res = await fetch(`${API_BASE}/api/correct`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ text }),
    });
    const data = await res.json();
    if (data.corrected) {
      voiceText.value = data.corrected;
      correctBtn.textContent = '✅ 已校正';
    } else {
      correctBtn.textContent = '❌ 校正失敗';
    }
  } catch {
    correctBtn.textContent = '❌ 校正失敗';
  }
  
  // 3 秒後恢復按鈕（可以再按一次微調）
  setTimeout(() => {
    correctBtn.disabled = false;
    correctBtn.textContent = '🔧 AI 校正';
  }, 3000);
});
```

⚠ 注意：AI 校正不像 AI 精簡那樣只能按一次。校正可以按多次（每次微調），所以**不鎖死按鈕**，3 秒後恢復。

### 🔵 DONE-60c：AI 校正模型逐一實測 → 選 Llama 3.3 ✅ 已部署 + 線上驗證

**用了哪個模型：`@cf/meta/llama-3.3-70b-instruct-fp8-fast`**（= 與 AI 精簡同款）。

**做法**：暫時讓 `/api/correct` 吃 `model` 覆寫參數、部署一次，用**同一段中文 + Python 正確送 UTF-8 + 檢視原始 bytes/code points**（不用會亂碼的 Git-Bash curl）逐一實測，再把贏家寫死、移除測試參數、最終部署。

**實測結果（輸入「呃那個我按了那個按鈕他就噴五白然後那個控制台有一個台破error」）**：
| 模型 | 結果 |
|---|---|
| `@cf/qwen/qwq-32b` | ❌ 不可用——輸出 935 字的冗長推理（且無 `<think>` 標籤可剝），贅字沒移除 |
| `@cf/deepseek/deepseek-r1-distill-qwen-32b` | ❌ HTTP 500 `5007: No such model`（此帳號沒這模型） |
| `@cf/qwen/qwen3-30b-a3b-fp8` | ✅ 乾淨正確（41 字，含 500/TypeError、去贅字、無亂碼） |
| `@cf/meta/llama-3.3-70b-instruct-fp8-fast` | ✅ 乾淨正確（33 字，同樣全對） |

**結論**：qwen3 與 llama-3.3 **都回乾淨正確的繁中**（再加 3 樣本複測皆過）。**這證實 DONE-60b 的判斷——先前看到的「Qwen3 亂碼」其實是 Windows Git-Bash 測試環境的編碼坑（request body 被送成非 UTF-8 + 終端機顯示中文花掉），不是 server；換模型本來就治不了那個顯示問題。**

**選 Llama 3.3 的理由**：兩者品質相當（qwen3 略勝在多去一個「那個」，差異極小），但 Llama 3.3 是**非推理模型**——無 `<think>` 額外 token 開銷、無 qwq 那種「推理外洩」風險、與 `/api/summarize` 同款（單一模型省心）、4 樣本實測穩定。保留 `<think>` 剝除（無害，日後換推理模型也安全）。

**收尾**：移除測試用的 `model` 覆寫參數與回應的 `model` 欄；`DEFAULT` 直接寫死 llama-3.3。

**tsc / deploy**：`tsc` ✅ + `wrangler deploy` ✅（測試版 `866413e7` → 最終版 **`e4729481`**）。最終生產端點 Python UTF-8 複驗：raw bytes 合法 UTF-8（`\xe6\x88\x91`=我）、500✓/TypeError✓/去贅字✓/無亂碼✓、`model` 欄已移除。

**待 FOX 驗收**：編輯頁實機按「🔧 AI 校正」（extension 是 UTF-8 fetch，無 Git-Bash 編碼坑）應看到乾淨校正。**若你之前是用 Windows 終端機 curl 測，看到的亂碼是終端機顯示問題，不是壞掉**——可改用瀏覽器或擴充實測。

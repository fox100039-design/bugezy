📌 關於產品

Q: BugEzy 是什麼？
A: BugEzy 是一款 Chrome 擴充功能，讓開發者用語音 + 錄製的方式記錄 Bug，AI 透過 MCP 自動讀取報告並提供修復建議。省下 95% 的 debug 溝通時間。

Q: 跟 Jam 有什麼不同？
A: BugEzy 專為亞洲開發者設計，支援中文語音辨識。價格只有 Jam 的 1/5（NT$80 vs $14 USD）。獨家功能：即時監控、30 秒回溯、終端機 CLI、Token 透明度。

Q: 支援哪些 AI 工具？
A: 任何支援 MCP 的 AI 工具都能用，包括 Claude Desktop、Claude Code、Cursor、VS Code + Copilot、Zed、Windsurf、Codex、Replit 等。只需要一行 URL：https://bugezy-api.bugezy-api.workers.dev/mcp

━━━━━━━━━━━━━━━━━━━━━━━━━━━

🔒 關於隱私與安全

Q: BugEzy 會錄到我的密碼嗎？
A: BugEzy 錄製的是 DOM 結構變化，不是螢幕截圖。密碼輸入框（type="password"）的內容會被 rrweb 自動遮蔽，不會錄到實際密碼。

Q: 我的報告誰能看到？
A: 只有你自己。報告預設為私人，只有當你主動分享報告連結時，別人才能看到。

Q: 資料存在哪裡？
A: 報告存在 Cloudflare R2（全球 CDN），使用者資料存在 Supabase（PostgreSQL）。所有傳輸都經過 HTTPS 加密。

━━━━━━━━━━━━━━━━━━━━━━━━━━━

💰 關於方案與付費

Q: 免費版有什麼限制？
A: 免費版每月可錄製 10 次、回溯 5 次、MCP 查詢 20 次。截圖標注和即時監控無限使用。報告保留 7 天。

Q: 付費版多少錢？
A: NT$80/月（約 $3 USD），解鎖全功能無限次使用，報告保留 90 天，加上終端機 CLI、Whisper 精準語音等進階功能。

Q: 如何升級付費版？
A: 在 BugEzy popup 點「升級」按鈕，透過信用卡或 ATM 付款。

Q: 可以取消訂閱嗎？
A: 可以，隨時取消。取消後當月剩餘天數仍可使用付費功能，下個月恢復為免費版。

━━━━━━━━━━━━━━━━━━━━━━━━━━━

🛠 關於技術

Q: 哪些瀏覽器支援？
A: 目前支援 Chrome 和所有 Chromium 瀏覽器（Edge、Brave、Arc 等）。

Q: 會影響網頁效能嗎？
A: 影響極小。BugEzy 只在你主動錄製時才記錄 DOM 變化，即時監控模式只攔截 Console error 和 Network error，不錄 DOM。

Q: MCP 是什麼？
A: Model Context Protocol（模型上下文協議），是 Anthropic 推出的開放標準，讓 AI 工具可以連接外部服務。BugEzy 的 MCP 讓 AI 直接讀取你的 Bug 報告，不需要複製貼上。

Q: Token 是什麼？為什麼 BugEzy 能省 Token？
A: Token 是 AI 處理文字的計量單位，等於你的 AI 使用費用。BugEzy 用結構化文字（而非截圖）傳送報告給 AI，同樣的 Bug 資訊只需要 1/20 的 Token。每次 MCP 查詢都會顯示 Token 估算，讓你看到省了多少。

document.querySelectorAll('.faq-q').forEach(q => {

### 🔵 DONE-68：跨頁回放滑鼠游標修復 ✅ build 通過（待 FOX 實機驗收）

**調查結論**：
1. **`mouseTail` 是 Replayer（回放）選項，不是 `record()` 選項**——故 inject.ts 兩處 `record()`（PM-50 背景緩存 line 102、錄製 line 585）都「正確地」沒有 `mouseTail`，不需也不該加。
2. **rrweb 預設就會錄 `mousemove`**（sampling.mousemove 預設開），所以跨頁每一段的游標資料其實「有錄到」，inject.ts 端無 bug、不需改。
3. edit-report.ts 的 Replayer 設定 **`mouseTail: true` 已正確套用**（line 142）。
4. **真正缺口（規格步驟 5）**：跨頁錄製時，每換一頁 rrweb 送一段新的 `FullSnapshot`，Replayer 重建 DOM；新頁在使用者「下一次移動滑鼠」前沒有 MouseMove 事件 → `.replayer-mouse` 游標在每個跳轉後段落開頭「消失」直到下次移動。

**修復**（只改 `extension/src/edit-report.ts`，inject/content/background 不動）：
- 新增 `injectCrossPageCursor(events)` 前處理：掃描事件流追蹤最後已知滑鼠座標（MouseMove `positions` 末筆 / MouseInteraction `x,y`），在**每個「非首段」`FullSnapshot`（type 2）後**補一筆**合成 MouseMove**（`type:3 / source:1 / positions:[{x,y,id,timeOffset:0}]`、`timestamp = 該段 FullSnapshot+1`），讓游標一進新頁就以上段最後座標立即現形。
- hover 目標 id 由 `findHtmlNodeId()` 從該 FullSnapshot 的序列化 DOM 取新頁的 `<html>` 節點 id（取不到則退而取任一元素節點；都沒有就跳過該段不注入，安全降級）——**必須用新頁實際存在的節點 id**，否則 rrweb `moveAndHover` 找不到節點會靜默 no-op（游標仍不動）。
- `initMiniPlayer()` 在 `new Replayer()` 前呼叫 `events = injectCrossPageCursor(events)`；首段不補（避免動到原始第一幀）。合成事件只新增 type 3，不影響 PM-38 用 type 4 Meta 算縮放、也不影響時長。

**型別**：自訂 `RREvent`/`RRPos` 載入型別（無 `any`），`tsc --noEmit` ✅ + `npm run build` ✅（`edit-report.js` 423.8kb）。

**🟡 提醒 FOX（需實機驗收）**：① 此修復針對「跳轉後段落開頭游標不見」。請依規格測試方式驗收：測試專頁錄製 → 跳測試頁 2 → 頁 2 移動滑鼠 → 停止 → 編輯頁回放看頁 2 段落游標。② 若游標在「同一段中途」也閃失（非段落開頭），那是另一個 PM-47 縮放 / `.replayer-wrapper` 重建的議題，需另開 PM；本 PM 僅處理段落邊界初始化。③ 此修復同步惠及線上 `/report/:id` 嗎？否——該頁是 server 端獨立 HTML，未走 edit-report.ts；若要同效果需另在 `REPORT_PAGE_HTML` 加同樣前處理（建議列入待辦）。

### 🟡 PM-69：打磨 — CSP 網站 inject 失敗

**問題**：某些 Content Security Policy 嚴格的網站（如 GitHub、某些銀行網站），inject.ts 被 CSP 擋住，BugEzy 無法錄製。

**可能原因**：
- inject.ts 透過 content.ts 用 `<script>` 標籤注入到 MAIN world
- 嚴格 CSP 不允許 inline script 或非白名單的 script

**調查 + 修復步驟**：

1. 讀 `extension/src/content.ts` 找注入 inject.ts 的方式
2. 如果是用 `document.createElement('script')` + `src`，改用 Chrome MV3 的 `chrome.scripting.executeScript({ world: 'MAIN' })` 方式注入
3. 在 manifest.json 確認 `"world": "MAIN"` 有在 content_scripts 設定
4. 測試在 GitHub.com 上能否正常錄製

**注意**：有些網站的 CSP 真的無法繞過，這是瀏覽器安全限制。對於這些網站，可以 graceful 降級（console 攔截還能用，但 rrweb DOM 錄製可能失敗），在 popup 提示「此網站部分功能受限」。

### 🔵 DONE-69：CSP 網站相容性 ✅ build 通過（待 FOX 在 GitHub 實機驗收）

**調查結論（重要：規格前提不成立）**：
- **content.ts 根本沒有用 `<script>` 標籤注入 inject.ts**——全 `extension/src` 無 `document.createElement('script')`、無 `executeScript`、無 `eval`/`new Function`。
- inject.ts 是**透過 manifest `content_scripts` 宣告式注入、且已設 `"world": "MAIN"`**（manifest.json 第 26–31 行，`run_at: document_start`）。**宣告式 content script 由瀏覽器擴充機制注入，本來就不受頁面 CSP `script-src` 限制**——這正是規格步驟 2/3 想「改用」的 CSP 相容方式，現況已是更好的版本。
- 故規格步驟 2（改用 `chrome.scripting.executeScript({world:'MAIN'})`）**不執行**——那反而是退步（executeScript 需 activeTab/時序處理、且 document_start 每個 frame 的可靠注入、PM-37 READY 握手都依賴宣告式注入）。規格步驟 2 本身是條件句「**如果**是用 script 標籤…」，條件不成立。

**真正的 CSP 缺口（已修）**：grep 找到 inject.ts 兩處在**頁面 MAIN world** assign `innerHTML`——在啟用 **Trusted Types** 的 CSP 網站（如 **GitHub** 的 `require-trusted-types-for 'script'`）assign `innerHTML` 會**直接拋例外**，這才是 BugEzy 在 GitHub 可能出問題的真因：
- **inject.ts:248**（`showCaptionBar` 的語音面板 header）——錄製中就會建，最關鍵。
- **inject.ts:200**（`toggleErrorPanel` 即時監控錯誤清單）。
- （`edit-report.ts:113`、`popup.ts` 的 innerHTML/src 在**擴充頁**執行，不受頁面 CSP 約束 → 不需改。）

**修復**（只改 `extension/src/inject.ts`，content/manifest/其他不動）：
- 語音面板 header 改用 `createElement('span') + textContent + appendChild` 取代 `header.innerHTML`。
- `toggleErrorPanel` 改用 DOM 節點建構（`appendRow()` 工廠：彩色標記 span + 內容 span，內容走 `textContent` 自動轉義），不再拼 HTML 字串 `panel.innerHTML`。
- 連帶移除原 `escapeHtml()`（只服務舊 innerHTML 拼字串；改 textContent 後本身即防注入，且 `noUnusedLocals` 會擋未使用函式）。

**驗證**：`tsc --noEmit` ✅ + `npm run build` ✅（`inject.js` 417.1kb）。

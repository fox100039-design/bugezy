### 🔵 DONE-75b：修 popup 付費用戶升級提示未隱藏 ✅ build 通過

**根因（CSS 特異度/順序 bug，非 JS）**：`popup.html` 裡 `.hidden { display:none }`（line 78）定義在 `.upgrade-hint { display:flex }`、`.manage-sub { display:flex }` **之前**。三者都是單一 class 選擇器（特異度相同），CSS 同特異度時**後定義者勝** → 帶 `class="upgrade-hint hidden"` 的元素仍套到 `display:flex`、**`hidden` 失效**。所以 PM-75 的 `loadPlan` 雖正確 `upgradeHint.classList.add('hidden')`，畫面上升級提示仍蓋掉、付費用戶同時看到「✨ 付費版 + 取消訂閱」和「升級付費版」。

**修法（CSS 層統一收斂，一處解決）**：`.hidden { display: none !important; }`——讓 `.hidden` 永遠勝過任何 `display:flex/block`，不受定義順序/特異度影響。一次修好 `upgradeHint`、`manageSubscription` 及所有用 `.hidden` 的元素（loginView/mainView/idle·recording·doneView 本來就靠 `.hidden`，加 `!important` 不影響其正確行為）。

**JS 端確認已收斂**：`upgradeHint` 顯示/隱藏只在兩處——① `loadPlan()`（plan 狀態權威來源：paid/cancelled `add('hidden')`、free `remove('hidden')`）② `startBtn` 的 `limitReached` 路徑（免費版用滿額度的執行期觸發；伺服器只對非付費回 limit_reached，付費不會誤觸）。兩者語意不重疊，無需再合併。

**Build**：`npm run build` ✅ + `tsc --noEmit` ✅；確認 `dist/popup.html` 含 `display: none !important`。

**🟡 提醒 FOX**：① 這是真正的 root cause（PM-75 改 JS 是對的，但被 CSS 蓋掉所以沒效果）。② **本 PM 只 build、未重打包 zip**（依規格）——重新上傳 Chrome Web Store 前請先重打包 `bugezy-v0.1.0.zip`（要的話我可代勞），目前磁碟上的 zip 仍是 PM-76 版（不含此 CSS 修正）。③ 實機驗收：付費/已取消用戶開 popup 應只見「✨ 付費版/已取消」卡片，**不再**有升級提示。

### 🟡 PM-78：MCP 報告過濾（只讀自己的）

**問題**：目前 MCP `list_reports` 回傳所有人的報告，任何人都能看。上線前必須修。

**修法**：MCP tools 加 `user_email` 參數，只回傳該 email 的報告。不帶 email 則回空（安全預設）。

---

#### Server MCP tools 改動

**list_reports**：
```ts
server.tool('list_reports', '列出 Bug 報告（需提供 email 過濾）', {
  user_email: z.string().optional().describe('使用者 email，只回傳該使用者的報告'),
  limit: z.number().optional(),
}, async ({ user_email, limit }) => {
  let query = supabase()
    .from('reports')
    .select('report_id, url, title, ...')
    .order('created_at', { ascending: false })
    .limit(limit || 10);

  if (user_email) {
    // 先查 user_id
    const { data: user } = await supabase()
      .from('users')
      .select('user_id')
      .eq('email', user_email)
      .maybeSingle();
    
    if (user) {
      query = query.eq('user_id', user.user_id);
    } else {
      return txtWithTokens([], 'list_reports');
    }
  } else {
    // 不帶 email → 回空（安全預設）
    return txtWithTokens(
      { message: '請提供 user_email 參數以查詢你的報告。例如：list_reports(user_email: "you@example.com")' },
      'list_reports'
    );
  }

  const { data } = await query;
  return txtWithTokens(data ?? [], 'list_reports');
});
```

**其他 tools（get_report_overview、get_console_logs 等）**：
這些用 report_id 查詢，不需要改。因為 report_id 是 UUID，不可能猜到別人的。只要 list_reports 過濾了，其他 tools 就安全了。

- extension / cli：不動
- get_report_overview、get_console_logs 等用 report_id 的 tools：不動

### 🟡 PM-79：Cron 保活 Supabase（防 7 天閒置暫停）

**問題**：Supabase 免費版 7 天沒請求會自動暫停資料庫。

**修法**：用 Cloudflare Workers Cron Trigger，每天自動 ping Supabase。

---

#### wrangler.toml 加 Cron

```toml
[triggers]
crons = ["0 3 * * *"]  # 每天凌晨 3:00 UTC
```

---

#### Server 加 scheduled handler

在 Worker 的 export default 裡加 `scheduled` 方法：

```ts
export default {
  async fetch(request: Request, env: Env, ctx: ExecutionContext) {
    // ...現有的 fetch handler...
  },

  async scheduled(event: ScheduledEvent, env: Env, ctx: ExecutionContext) {
    // 每天 ping Supabase 保活
    try {
      const supabase = supa(env);
      const { count } = await supabase
        .from('users')
        .select('user_id', { count: 'exact', head: true });
      
      console.log(`[Cron] Supabase keepalive: ${count} users`);
    } catch (err) {
      console.error('[Cron] Supabase keepalive failed:', err);
    }
  },
};
```

⚠ 注意：如果目前的 export default 是用 class（McpAgent），需要把 scheduled 加在正確的位置。Claude Code 請檢查現有的 export 結構再決定怎麼加。
